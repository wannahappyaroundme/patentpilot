## 3. 문제 정의

### 3.1 시장 현실

- 한국 IP 정보서비스 시장 = **연 1조 1,311억** (미국의 1/10, 일본의 1/2)
- 한국 정부 R&D 투자는 GDP 대비 세계 최상위 수준
- 그러나 정출연·대학에서 나온 특허의 **사업화율은 한 자릿수**로 추정
- 기업이 한국 공공특허를 *찾을 도구* 자체가 부족
- KIPRIS는 검색은 가능하지만, **관계 기반 탐색·매칭**은 불가

### 3.2 이해관계자별 페인포인트

| 이해관계자 | 페인포인트 |
|---|---|
| 기업 R&D 팀 | "우리 문제 해결 기술이 한국에 있는지 모름. 글로벌 검색만 함" |
| TLO/산학협력단 | "보유 특허를 누구한테 팔아야 할지 모름. 매칭 채널 없음" |
| 교수·연구원 | "내 특허 등록 후 뭘 해야 하는지 모름. 출원이 끝" |
| VC/투자자 | "스타트업 기술력 검증할 객관적 지표 부족" |
| 정책연구자 | "정부 R&D가 정말 사업화로 이어졌는지 측정 어려움" |

### 3.3 시장 안의 *비어 있는 자리*

- WIPSON, PatSnap = 글로벌 폐쇄형, 한국 R&D 특화 약함
- 키프리스 = 검색 중심, 매칭·관계 분석 없음
- 11년 대회 수상작 = 정출연·대학 특화 사례 **0건**
- → **TreeO가 정확히 이 빈자리에 들어감**

## 4. 솔루션 — TreeO

### 4.1 2층 구조 (Open Core)

```
┌─────────────────────────────────────────────────────┐
│  Layer 2: PRODUCT (유료, B2B SaaS)                  │
│  ─────────────────────────────────────────────────  │
│  ⓐ K-R&D Patent Match  (기업 R&D, TLO 대상)         │
│  ⓑ K-R&D Insight       (TLO 대시보드)               │
│  ⓒ K-R&D Trend         (VC/금융기관 투자 레이더)     │
│  ⓓ K-R&D Whitespace    (R&D 기획자 화이트스페이스)   │
└──────────────────┬──────────────────────────────────┘
                   │ uses
┌──────────────────▼──────────────────────────────────┐
│  Layer 1: FOUNDATION (무료, 오픈소스)                │
│  ─────────────────────────────────────────────────  │
│  • TreeO Graph     : 정출연·대학 37만 특허 그래프 DB │
│  • TreeO Score     : 다차원 스코어링 엔진 (5축)     │
│  • TreeO API       : 누구나 호출 가능 Open API       │
│  • TreeO Ontology  : OWL 기반 표준 어휘              │
└─────────────────────────────────────────────────────┘
```

### 4.2 핵심 가치 명제

- **공급(특허) ↔ 수요(기업 문제)를 거미줄로 잇는다**
- **자연어로 묻기만 하면, 그래프가 답을 거슬러 올라가서 보여준다**
- **무료 인프라로 한국 IP 시장을 키우고, 유료 제품으로 지속가능성을 확보한다**

### 4.3 대회용 메인 출품 = ⓐ K-R&D Patent Match

- 다른 3개(ⓑⓒⓓ)는 *동일한 그래프에서 파생되는 자연스러운 확장*임을 강조
- 심사위원에게는 "1개 제품 + 4개 확장 + 1개 인프라" = 1석 6조 어필

## 5. 사용자 인터페이스 (UX) — 대화형

### 5.1 컨셉: "한국 R&D 특허 전문 ChatGPT"

- 검색창이 아니라 **채팅창**
- 사용자는 자연어로 질문, TreeO는 그래프를 자동 탐색해서 답
- 한 번의 답으로 끝나지 않고, **꼬리에 꼬리를 무는 대화형 탐색**
- 텍스트 + 표 + 그래프 시각화 + 차트가 함께 출력 (멀티모달)
- 그래프 노드 클릭 시 즉시 해당 노드로 대화 컨텍스트 이동

### 5.2 대화 시나리오 예시 (페르소나: 기업 R&D 박팀장)

```
[A] 박팀장: "우리는 전기차 배터리 발열을 줄이는 기술이 필요해"

TreeO:  📊 매칭된 한국 R&D 특허 12건 (관련도 높은 순)
        ┌─────────────────────────────────────────┐
        │ 1. KAIST | 김OO | 2022 | 종합 87점       │
        │ 2. KIST  | 이OO | 2021 | 종합 84점       │
        │ ...                                      │
        └─────────────────────────────────────────┘
        [📈 매칭 그래프 시각화]
        💡 각 특허 클릭 시 상세 정보

[B] 박팀장: "1번 특허 자세히"

TreeO:  📄 [특허 상세]
        - 발명자: 김OO 교수 외 3인
        - 권리자: KAIST 산학협력단
        - 청구항 12개 (독립항 3, 종속항 9)
        - 인용된 특허 28건 / 인용한 후속 특허 51건
        - 패밀리: 미국, 일본, 유럽 출원
        - 권리 잔여기간: 14년
        - 기술이전 이력: 없음 (이전 가능 ✅)
        
        🕸️ [발명자·기관·인용 거미줄 시각화]

[C] 박팀장: "이 특허를 살 만한 다른 기업 비교군 알려줘"

TreeO:  🏢 유사 기술 도입한 기업 5곳 (그래프 인접 노드)
        - LG에너지솔루션 (인용 8건)
        - 삼성SDI (인용 5건, 자체 특허 12건)
        - SK온, 포스코퓨처엠, 에코프로비엠
        💡 이전 가능성 점수, 컨택 우선순위 추천

[D] 박팀장: "김OO 교수 다른 특허도 보고 싶어"

TreeO:  👤 김OO 교수 보유 특허 23건
        - 분야 분포: 배터리(15), 연료전지(5), 기타(3)
        - 주요 공동 연구자: 박OO(KAIST), 이OO(POSTECH)
        - 협력 기관: LG화학(2건), SK이노베이션(1건)
        🕸️ [김교수 중심 거미줄 그래프]
        
[E] 박팀장: "이 분야 한국 핵심 연구자 Top 5는?"

TreeO:  🏆 발명자 허브 스코어 Top 5 (배터리 발열 분야)
        ...
```

### 5.3 화면 구성 요소

- **좌측**: 대화 히스토리
- **중앙**: 채팅창 + 답변 영역 (텍스트 + 표 + 차트)
- **우측**: 그래프 시각화 패널 (실시간 거미줄, 노드 클릭 인터랙션)
- **상단**: 페르소나 선택 (5개 모드 전환), 데이터 필터(연도, 기관, IPC)
- **하단**: 추천 후속 질문 ("이런 것도 물어보세요")

### 5.4 UX 원칙 5가지

1. **검색 버튼 없음** — 모든 입력은 대화
2. **그래프는 항상 보인다** — 답변과 함께 시각화 동기화
3. **답변에 출처가 있다** — 클릭 가능한 특허 ID, 발명자, 기관 링크
4. **한 답변에 끝나지 않는다** — 항상 후속 질문 제안
5. **시각이 곧 탐색이다** — 그래프 노드 클릭 = 새 대화 시작

| 발명자 식별자 | 연구자 ORCID, 학력 | Crossref, OpenAlex | ★★ |
| 기업 정보 | 매출, 업종, R&D 규모 | DART, KISLINE | ★★ |
| 글로벌 패밀리 | 한국 특허의 USPTO/EPO 출원 | PatentsView, EPO OPS | ★ |
| 기술이전 사례 | 과거 라이선싱 거래 | 한국 TLO 협회, 정부 통계 | ★ |
| 비특허 인용 | 논문 인용 | Crossref, OpenAlex | ★ |

### 7.3 데이터 정제·정규화 정책

#### 발명자 동명이인 처리

- **동일 이름 + 동일 기관** = 일단 동일인 추정
- **동일 이름 + 다른 기관** = 별개 ID 부여, 단 시간상 이력 일관 시 동일인 검토
- **유사 표기(오탈자) + 동일 기관** = 사람 검토 큐로 보냄
- 모든 판정에 **신뢰도 0~1** 부여, 임계값 이상만 자동 병합
- 추후 머지/스플릿 가능한 구조

#### 기관명 표준화

- 표준형(canonical) 1개 + 변형(variants) 다수 저장
- 예: `kaist` ← {KAIST, 한국과학기술원, Korea Advanced Institute of Science and Technology}
- 산학협력단과 본교 분리 vs 통합 정책 결정 필요 (1주차 결정 사항)
- 합병/분리 이력 시간축으로 관리

#### 결측값 처리

- 누락 시 NULL 허용 + 플래그 처리
- 자동 보완(추론) 시 출처 명시 (PROV-O 패턴)
- 검증 안 된 추론값은 별도 신뢰도 표시

### 7.4 데이터 라이선스 (가장 중요한 사전 점검)

- **즉시 행동 필요**: KIPRIS Plus 라이선스 담당자에게 *재배포 가능 여부* 공식 문의
- KIPI 라이선스 담당 (연락처 비공개)
- 답변에 따라 시나리오 분기:
  - (A) 재배포 가능(라이선스 확정 시) → Foundation 데이터 단계적 공개
  - (B) 재배포 제한 → Foundation은 어휘·스코어링·코드만 오픈, 데이터는 회원제
  - (C) 학술 한정 → 학계 사용자에게만 오픈, 비학계는 API 호출

| `University` | 대학 | 국립/사립/특수 |

| `RDProject` | NTIS R&D 과제 | 과제번호, 부처 |

### 8.2 엣지 (Relationship Types)

| 관계 | 의미 | 방향 |
|---|---|---|
| `inventedBy` | 특허 → 발명자 | 단방향 |
| `ownedBy` | 특허 → 권리자 (시점 의존) | 단방향 |
| `affiliatedWith` | 사람 → 기관 (시점 의존) | 단방향 |
| `cites` | 특허 → 특허 (인용) | 단방향, 비대칭 |
| `familyMemberOf` | 특허 ↔ 특허 (패밀리) | 양방향, 대칭, 전이 |
| `classifiedAs` | 특허 → IPC | 단방향 (주/부 구분) |
| `coInventedWith` | 발명자 ↔ 발명자 | 양방향, 대칭 |
| `collaboratesWith` | 기관 ↔ 기관 | 양방향, 대칭 |
| `transferredTo` | 특허 → 새 권리자 (이벤트) | 단방향 |
| `derivedFrom` | 특허 → R&D 과제 | 단방향 |
| `subjectOfTrial` | 특허 → 심판 | 단방향 |

### 8.3 다차원 스코어링 (5축)

| 스코어 축 | 정의 | 그래프 활용 |
|---|---|---|
| **기술 영향력** | 후속 기술에 미친 영향 | 인용 그래프의 PageRank |
| **권리 강도** | 청구항 폭, 패밀리 규모 | 청구항 노드 분석 |
| **시장 가치** | 분쟁·거래·표준 이력 | Event 노드 가중 |
| **발명자 허브력** | 협력 네트워크 중심성 | 공동발명 그래프의 Betweenness |
| **기관 협력력** | 기관 간 협력 다양성 | 기관 그래프의 Eigenvector Centrality |

→ 5축을 통합한 **종합 스코어**(Composite Score)도 제공하지만, 가중치는 사용자가 조정 가능 (Open Core의 핵심 가치)

### 8.4 그래프 모델링 패턴 결정 (1주차 합의 사항)

- **Role 패턴**: Reification 채택 (Person → InventorRole → Patent)
- **Event 패턴**: Event Reification (모든 이벤트는 노드)
- **Time 모델**: Event-sourcing (라이프사이클 = 이벤트 시퀀스)
- **Identity**: Confidence-scored, mergeable

## 9. AI/LLM 전략

### 9.1 Architecture: RAG on Graph

```
[사용자 자연어 질문]
       ↓
[1] Intent Detection (LLM)
   "이건 매칭 질문" / "이건 발명자 질문" / "이건 트렌드 질문"
       ↓
[2] Graph Query Planning
   필요한 노드, 엣지, 필터 결정
       ↓
[3] Graph Traversal (Neo4j / SPARQL)
   실제 그래프 탐색 + 스코어 적용
       ↓
[4] Result Ranking
   다축 스코어로 정렬
       ↓
[5] Response Synthesis (LLM)
   결과 → 자연어 답변 + 차트 + 시각화 데이터
       ↓
[사용자에게 표시]
```

### 9.2 모델 선택

| 용도 | 모델 후보 | 비고 |
|---|---|---|
| 한국어 특허 임베딩 | **KorPatBERT** (KIPI 공개) | 도메인 특화, 무료 |
| 청구항 임베딩 | PaECTER, PatentSBERTa | 영문, 한국어 적용 검토 |
| Intent / Synthesis LLM | Llama 3.1, Qwen 2.5, Polyglot-Ko | LoRA fine-tune |
| 백업 LLM (개발 초기) | OpenAI GPT-4o, Claude API | 빠른 프로토타입 |

### 9.3 학습/Fine-tune 전략

- **Phase 1 (대회용)**: KorPatBERT + GPT/Claude API → 시연 가능 수준
- **Phase 2 (수상 후)**: Llama/Qwen LoRA fine-tune (정출연·대학 특허 데이터로)
- **Phase 3 (1년 후)**: 자체 학습 데이터 축적 → 도메인 특화 모델 자체 보유

### 9.4 학습 데이터 확보 정책

- 정답 셋 (Entity Resolution용): 수동 라벨링 + 산학협력단 협조
- 동의어 사전: 기관명, 발명자명 표준화 작업의 부산물
- 평가 셋: 학계 표준 벤치마크 + 자체 Gold Standard

| 발명자/기관 네트워크 | 약함 | 1차 상품 |

### 11.2 vs 11년 수상작 분석

| 영역 | 과거 수상작 | TreeO 차별 |
|---|---|---|
| 검색 (1~3회) | 키워드 검색 | 자연어 + 그래프 탐색 |
| 분류 (4~6회) | 딥러닝 분류 | 분류 + 관계 추론 |

| 가치평가 (9회) | 단일 점수 | 다축 + 그래프 가중 |
| 자동생성 (10회) | 선행조사 보고서 | 매칭 + 컨택 추천 |
| **정출연·대학 특화** | **0건** | **TreeO 단독** |
| **오픈소스** | **거의 0** | **TreeO 강조** |

### 11.3 차별점 5가지 (대회 발표 핵심 메시지)

1. **데이터의 차별** — 정출연·대학 37만 건 정밀 큐레이션 (남이 못 가짐)
2. **구조의 차별** — 그래프 + 다차원 스코어 (단순 DB가 아닌 관계 인프라)
3. **인터페이스의 차별** — 검색 아닌 대화 + 시각 (사용자 학습 비용 0)
4. **모델의 차별** — Open Core (사회적 가치 + 수익 모델 동시)
5. **비전의 차별** — 한국판 PatentsView (단일 제품 아닌 인프라)

### 19.3 FAQ

**Q: 왜 그래프 DB여야 하나요? 관계형으로 안 되나요?**
A: 가능합니다. 다만 *N-hop 탐색*("이 발명자의 협력 기관의 다른 특허")이 관계형에서는 JOIN 폭증으로 느려집니다. 그래프 DB는 이걸 자연스럽게 처리합니다.

**Q: 오픈소스로 공개하면 카피당하지 않나요?**
A: 데이터 정제·매칭 정확도·UX가 진짜 자산입니다. Foundation은 미끼이고, Product가 해자입니다. MongoDB, Hugging Face가 같은 모델로 성공했습니다.

**Q: 1달 안에 다 만들 수 있나요?**
A: 1달은 *대회 출품용 시제품 + 기획서*까지입니다. 진짜 시스템은 수상 이후 7년에 걸쳐 만듭니다. 1달 목표는 "대상 받기"이고, 그 후 인프라가 본격 시작됩니다.

**Q: KIPRIS 라이선스가 막히면 어떻게 하나요?**
A: 데이터는 회원제로 가고, 어휘·스코어링·코드만 오픈합니다. 핵심 메시지는 유지됩니다.

**Q: AI 모델은 자체 학습해야 하나요?**
A: 1단계는 KorPatBERT + 외부 LLM API. 2단계에서 자체 fine-tune. 처음부터 자체 학습은 비효율.
